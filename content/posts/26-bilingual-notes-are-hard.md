# 한영 Bilingual 노트의 어려움

meshblog를 만들면서 한국어와 영어를 섞어서 글을 쓰기로 결정했다. 기술 용어는 영어로, 설명과 생각은 한국어로. 자연스럽고 빠르다. 그런데 이게 NLP 파이프라인에서 파국을 일으킨다.

## 실제로 부닥친 문제들

### 검색이 잘 안 됨: Fuse.js + Hangul

처음엔 Fuse.js로 클라이언트 전문검색을 했다. 간단하고 빠르다. 그런데 한글 쿼리가 먹히지 않았다.

```
"리액트" 검색 → 0개 결과
"react" 검색 → 정상
```

왜? Fuse.js는 문자 거리(char distance) 기반이다. 한글은 자모 분해(초성, 중성, 종성)가 필요한데, Fuse.js는 이를 몰라서 "리액트" 문자열을 직접 찾으려 한다. 근데 마크다운에 "React"라고 있으면? 매치 안 됨.

### 임베딩 모델의 혼란: 한영 혼합 청크

meshblog의 Q&A 기능은 text chunks를 임베딩해서 유사도로 답변을 생성한다. 문제는 한 청크가 한글과 영어를 섞고 있다는 것.

```
"형태소 분석 사례: tokenizer performance는 한글에서 어렵다"
```

대부분의 임베딩 모델(OpenAI text-embedding-3, Jina v3)은 영어에 최적화돼 있다. 한글은... 잘하긴 하는데, 한영 혼합 청크의 의미 공간이 이상해진다. 한글과 영어 토큰이 embedding space에서 떨어져 있을 수 있기 때문이다.

실제로 한글만 있는 청크가 영어 쿼리로는 검색이 잘 안 되고, 반대도 마찬가지다.

### Pretendard 폰트 서브셋: 빌드 복잡도

한글을 제대로 렌더링하려면 가변폰트를 쓰거나 서브셋을 만들어야 한다. meshblog는 Pretendard를 썼는데, 한글 + 라틴 문자 모두를 커버하는 서브셋을 만드는 게 까다롭다.

폰트 파일 크기도 문제다. 한글 글리프는 2만개 이상이다. 전체를 포함하면 1MB가 넘는다. 서브셋으로 줄여도 300KB 정도.

### 형태소 분석의 필요성

한글 텍스트를 검색하거나 임베딩하려면 형태소 분석이 거의 필수다. "리액트 사용법"을 검색하려면:

1. "리액트" → 명사
2. "사용법" → 명사

그런데 한글을 형태소 분석 없이 n-gram으로 나누면? "리", "액", "트", "사", "용", "법" 같은 글자 단위로 쪼개진다. 검색이 엉망이 된다.

JavaScript에서 한글 형태소 분석을 제대로 하려면 라이브러리를 써야 한다. 근데 WASM 바이너리가 필요하고, 빌드 복잡도가 올라간다.

## 언어별로 나눠야 하나?

처음 생각은 "그냥 섞어서 쓰자"였는데, 실제로는:

- 한글 전용 페이지: Fuse.js + 간단한 한글 형태소 분석기 필요
- 영어 전용 페이지: Fuse.js 기본 그대로
- 한영 혼합: 임베딩 쿼리 전에 언어 감지 후 청크 필터링

이게 더 깔끔하다.

## 현재의 전략

### 1. 한글 URL Slugs

`/post/한영-bilingual-노트의-어려움` 대신 `/post/26-bilingual-notes-are-hard` 로 통일했다. 이유:

- 웹 표준은 ASCII 기반. 한글 URL은 인코딩 문제가 많다
- 검색 엔진이 한글 슬러그를 제대로 크롤링 안 할 수 있다
- 터미널에서 다루기 불편하다

대신 frontmatter title에 한글을 넣는다.

### 2. 청크별 언어 정보

임베딩 전에 청크마다 주 언어(한글/영어)를 태그한다. Q&A 쿼리가 들어오면 먼저 쿼리 언어를 감지한 후, 같은 언어의 청크들만 검색한다.

한글 쿼리 → 한글 청크만
영어 쿼리 → 영어 청크만

이렇게 하니 검색 품질이 올라갔다.

### 3. 형태소 분석 (선택적)

한글 쿼리에 대해 간단한 형태소 분석을 한다. 라이브러리로는 `kiwi.js` 같은 경량 버전을 썼는데, 빌드 크기가 문제면 서버 측 분석으로 옮긴다.

### 4. 폰트 최적화

Pretendard 대신 Google Fonts의 `Noto Sans KR` 또는 `Pretendard JP Variable`로 갈 생각 중이다. 가변폰트면 weight 조정이 유연하고, 서브셋도 크기가 더 작다.

아니면 시스템 폰트(SF Pro, -apple-system)에 `"Malgun Gothic"` 폴백을 추가하는 방법도 있다.

## 정리

bilingual 블로그는 보기에는 깔끔한데, 뒤에서는 검색, 임베딩, 폰트, URL 등 여러 부분에서 복잡도가 튀어나온다. 

현재 meshblog의 선택:
- URL: 영어 슬러그
- 제목/본문: 한영 혼합
- 검색: 언어별 분리
- 임베딩: 언어 태그 + 청크 필터링
- 폰트: 가변폰트 + 시스템 폴백

이게 완벽하진 않지만 실용적이다. 의도는 독자가 자연스러운 경험을 하는 것이고, 검색과 추천이 언어를 이해하는 것. 그걸 위해선 백엔드에서 좀 더 일해야 한다.
