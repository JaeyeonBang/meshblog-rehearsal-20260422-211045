# design-md-sync

`design.md` at project root is the single source of truth for all visual tokens. Edit it → regenerate `src/styles/tokens.css` → preview.

## When to invoke

- "change the accent color to X"
- "make the body text bigger"
- "tighten the spacing"
- "switch to a warmer paper tint"
- Any change to color, typography, space, radius, motion, shadow

## Steps

1. Read `design.md` — understand current frontmatter (colors, fonts, scale, motion, rules, shadows).
2. Read `design-ref/SPEC.md` §3 — confirm exact token names you must preserve.
3. Edit only the frontmatter. Body prose is descriptive, does not affect tokens. Keep all required keys (removing a key breaks the build).
4. Regenerate: `cd /mnt/d/projects/meshblog && bun run build-tokens`
5. Verify exit 0 and spot-check `src/styles/tokens.css` — banner says `AUTOGENERATED` and new values appear.
6. Preview: `bun run dev` → open `http://localhost:4321/`. Confirm the change is visible.

## Guardrails

- No hex literals outside `tokens.css` / `fonts.css`. Downstream files must reference `var(--ink)` etc.
- Do not rename token keys — components + pages reference them by name. Renaming `--ink` → `--black` breaks 25 components.
- Dark mode values under `colors.dark:` override light. Omit to inherit auto-invert.
- Never edit `src/styles/tokens.css` directly — autogenerated, changes lost on next build.

## Verify change reached the browser

```bash
curl -s http://localhost:4321/ | grep -o '--ink:[^;]*' | head -1
```

## Rollback

```bash
git checkout design.md && bun run build-tokens
```
