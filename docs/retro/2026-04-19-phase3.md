## 3. Drift Observed

Every plan-vs-reality mismatch found during implementation:

- **`MarkdownView` prop: `content` → `markdown`.** Plan Task 4 pseudocode used `content`; actual component uses `markdown` to avoid shadowing the JS `Element.content` API. Every Astro page passing content had to use `markdown={post.content}`.

- **`QAChips` prop: `cards` → `qas`.** Plan named it `cards`; implementation uses `qas` (aligns with `QaRow` type name). Scope value `"note"` → `"page"` — plan said `scope="note"` for per-page chips; implementation uses `"page"` because the same chip appears on both notes and posts.

- **Notes manifest shape: `Array<entry>` → `Record<id, entry>`.** Plan Task 3 pseudocode showed `manifest[n.id] = { ... }` (object), but the type comment said "Array". Final shape is `Record<string, ManifestEntry>` keyed by note id. GraphView and graph-json-manifest tests both use the Record form.

- **`PostRow` / `NoteRow` no `excerpt` field.** Plan index.astro excerpt in posts feed. No `excerpt` column in `notes` table. Computed at render time in `index.astro` by truncating `content` to 160 chars; not a helper-layer field.

- **`listPosts(db)` in plan → `listPosts()` in actual (no db arg).** The plan showed all helpers taking `db` as first argument. The actual `src/lib/pages/posts.ts` still takes `db` as an argument (matches plan). No drift here — confirmed by reading the implementation.

- **Base.astro creation conflict: Task 7' vs Task 8.** Both tasks specified creating `src/layouts/Base.astro`. Task 7' (OG/RSS) ran first and created it with `ogImage` prop support. Task 8 (index.astro) absorbed Base.astro into its own work without conflict — Task 7' write stood. The plan didn't flag this dependency; it resolved cleanly because tasks ran sequentially.

- **`qa_cards` schema: `scope_id` in plan → separate `note_id` + `concept_id` columns.** Several plan SQL snippets referenced a generic `scope_id`; the actual Phase 2 schema uses separate nullable `note_id` and `concept_id` FK columns. `getQaForNote` and `getQaForConceptsOfNote` query against `note_id` and `concept_id` respectively.

- **`getRelatedNotes` no `folder_path` in output → `manifest.json` resolution.** Plan Task 1 `RelatedNote` type includes `folder_path`. The actual implementation returns `folder_path` from the DB but uses it inline to compute `href` in the page files, rather than exposing it through the manifest. The GraphView's click routing goes through `public/notes-manifest.json` as designed (Patch C5 decision).

- **`build:fixture` doesn't build post pages.** The fixture `seed.sql` inserts notes under `folder_path = '/notes'` only; no fixture posts. `bun run build:fixture` therefore produces zero `dist/posts/` pages. Not a bug; fixture was designed for notes. The `build-smoke.test.ts` test was updated to check `at least one dist/notes/<slug>/` rather than hardcoded post slugs.

## 4. Decisions We'd Redo

**Keep QAChips scope as a string enum, not a boolean.** The plan wavered between `scope="note"` and `scope="page"`. We ended on `"page"` vs `"global"`. In hindsight, the distinction only matters for Fuse key weights; a boolean `isGlobal` or a `weightPreset` prop would have been cleaner and more honest.

**Extract concept-QA lookup into a helper, not inline in page files.** `getQaForConceptsOfNote` is a 10-line 3-way join inside `src/lib/pages/qa.ts`. Both `posts/[slug].astro` and `notes/[slug].astro` call it and then spread the results into `allQa = [...noteQa, ...conceptQa]`. That spread pattern is copy-pasted verbatim across two pages. A single `getQaForPage(db, noteId)` that returns the merged and deduped array would have removed the duplication at zero cost.

**Run build-smoke as an isolated integration test, not in the main `vitest run`.** The current setup puts `test/build-smoke.test.ts` inside the default test suite. When vitest runs all 16 test files in parallel, `build-smoke` triggers `bunx astro build` while `fixture-mode.test.ts` also calls `build-index` → both fight for `.data/index.db` → SQLITE_BUSY → 2 failures. The fix is a `test:integration` script with `--project build-smoke` run separately. Phase 4 should move it.

## 8. Phase 5 Pre-work Handoff

What the design sprint needs from the engineering side before Phase 5 starts:

**Token system:** No CSS variables or design tokens exist. Phase 5 will introduce a token layer (color, type scale, spacing). The current inline `style={{ fontSize: '16px' }}` in QAChips (AF5) is the only hardcoded design value that will need to migrate to a token. Everything else uses browser defaults.

**Graph aesthetic pass:** GraphView renders uniform gray circles and light gray lines. The `GraphNode.pagerank` and `GraphNode.type` fields are already in the data (`src/components/graph/types.ts`). Phase 5 needs to decide: pagerank → node radius scale, type (`note` vs `concept`) → stroke color. The hook (`useForceSimulation.ts`) draws circles with hardcoded `r=6` and `fill=#555` — both need to become configurable props.

**OG card design:** `scripts/build-og.ts` generates SVGs with a hardcoded layout: gray background, title in Inter 48px, site name in Inter 24px. The font is bundled as `Noto_Sans_KR` bytes in the script. Phase 5 needs to provide a design spec for the OG template (brand color, logo placement, typography). The script is already parameterized by title+description; only the SVG template string needs updating.

**Empty states:** Four empty states exist but have no visual design: QAChips "no FAQ yet" (`<p>` with `<em>`), GraphView loading/error/empty (plain text), index.astro "no posts" (`<section><p>`), related notes empty (silent). Phase 5 should spec illustrations or at minimum styled copy for each.

**T2-T5 taste decisions:** backlinks vs related (T2), node encoding (T3), tagline (T4), long-answer overflow (T5) — all need a user decision before Phase 5 implements them.
